# Card B on 2× RTX PRO 6000 — verified bench (2026-05-28)

This file documents the first end-to-end-on-fresh-Docker Card B serve and bench against the canonical `canada-quant/dsv4-rtx6000pro:v3` image. Replaces speculation about whether the recipe works on `mtcl`-class hardware (2× RTX PRO 6000 Blackwell Server Edition) with measurement.

## Config

- Hardware: 2× NVIDIA RTX PRO 6000 Blackwell Server Edition (SM 12.0, 96 GiB HBM each, no NVLink — PCIe)
- vLLM: `jasl/vllm@27fd665b` (`ds4-sm120-preview-dev`) + 5 patches + canada-quant cherry-pick + cute.arch.fmin shim + tuple_return forced-false
- Image: `canada-quant/dsv4-rtx6000pro:v3` (Docker, NGC PyTorch 26.04 base, 13 patch layers)
- Serve: TP=2, MTP k=1, max_model_len=8192, max_num_seqs=2, gpu_memory_utilization=0.95, cudagraph_capture_sizes=[1,2], `--enforce-eager` OFF
- Env: `VLLM_TEST_FORCE_FP8_MARLIN=1 VLLM_USE_LAYERNAME=0`

## Throughput sweep (`vllm bench serve`, random 256-in / 256-out)

| Concurrency | n | Avg out tok/s | Peak out tok/s | Median TPOT (ms) | MTP acceptance |
|---|---|---|---|---|---|
| **bs=1** | 4 | **45.14** | **73.00** | 8.13 | 71.4% |
| bs=2 | 8 | 21.13 | 26.00 | 80.04 | 73.7% |

Note: bs=2 scales poorly (per-stream throughput drops 4×) due to TP-allreduce communication over PCIe + Marlin BF16 MoE (no native FP4 on SM 12.0). **Optimal config for 2× RTX PRO 6000 is single-user (bs=1)**. For high-concurrency serving, use TP=4 on 4× cards or B200/B300.

## Thinking mode

⚠️ **Functional, with parser caveat** — the model emits the full thinking chain, but vLLM's `--reasoning-parser deepseek_v4` (with MTP enabled) writes the reasoning to `content` instead of `reasoning_content`. This is upstream issue [vllm-project/vllm#34650](https://github.com/vllm-project/vllm/issues/34650). The reasoning is preserved; the response just doesn't split into the two structured fields. For AIME-style evals where the final integer is extracted from anywhere in the response, this is invisible. For applications that explicitly read `reasoning_content`, set `chat_template_kwargs.thinking=false` or drop `--reasoning-parser`.

## What this verified for the first time

- Card B (NVFP4-FP8-MTP) loads and serves end-to-end on a fresh Docker container with no leftover host state
- The 13-patch Docker recipe (`docker/Dockerfile.rtx6000pro`) reproduces the `install_rtx6000pro_v3.sh` host install — the v3 script's "verified 2026-05-27" claim was a smoke test on a polluted host; this is the first clean-environment verification
- BF16 MTP block loads with `canada-quant/vllm@5a49d8803+5acabf315` cherry-pick (issue #43304 fix)
- Tool calling + thinking-mode + reasoning all functional on 2× RTX PRO 6000
- TP=2 with CUDA graphs ON is the optimal serve config for the 2-GPU mtcl-class deployment
