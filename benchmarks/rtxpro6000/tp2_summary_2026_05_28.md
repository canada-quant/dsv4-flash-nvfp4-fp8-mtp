# Card B on 2× RTX PRO 6000 — verified bench (2026-05-28)

This file documents the first end-to-end-on-fresh-Docker Card B serve and bench against the canonical `canada-quant/dsv4-rtx6000pro:v3` image. Replaces speculation about whether the recipe works on `mtcl`-class hardware (2× RTX PRO 6000 Blackwell Server Edition) with measurement.

## Config

- Hardware: 2× NVIDIA RTX PRO 6000 Blackwell Server Edition (SM 12.0, 96 GiB HBM each, no NVLink — PCIe)
- vLLM: `jasl/vllm@27fd665b` (`ds4-sm120-preview-dev`) + 5 patches + canada-quant cherry-pick + cute.arch.fmin shim + tuple_return forced-false
- Image: `canada-quant/dsv4-rtx6000pro:v3` (Docker, NGC PyTorch 26.04 base, 13 patch layers)
- Serve: TP=2, MTP k=1, max_model_len=8192, max_num_seqs=2, gpu_memory_utilization=0.95, cudagraph_capture_sizes=[1,2], `--enforce-eager` OFF
- Env: `VLLM_TEST_FORCE_FP8_MARLIN=1 VLLM_USE_LAYERNAME=0`

## Throughput sweep (`vllm bench serve`, random 256-in / 256-out)

| Concurrency | n | Avg out tok/s | Peak out tok/s | Median TPOT (ms) | MTP acceptance |
|---|---|---|---|---|---|
| **bs=1** | 4 | **45.14** | **73.00** | 8.13 | 71.4% |
| bs=2 | 8 | 21.13 | 26.00 | 80.04 | 73.7% |

Note: bs=2 scales poorly (per-stream throughput drops 4×) due to TP-allreduce communication over PCIe + Marlin BF16 MoE (no native FP4 on SM 12.0). **Optimal config for 2× RTX PRO 6000 is single-user (bs=1)**. For high-concurrency serving, use TP=4 on 4× cards or B200/B300.

## AIME-2024 full 30 (c=1, thinking=high, max_tokens=8000, pinned image)

Run after pin tightening + image rebuild — to settle whether the pinned image regresses anything.

| Metric | Pinned (this run) | Baseline 2026-05-25 |
|---|---|---|
| **MTP acceptance** | **90.46%** | **90.61%** ✓ matches within 0.15pt |
| Correct/30 | 17/30 (56.7%) | 24/30 (80%) |
| Errors | 2 | 0 |
| Finish: stop / length | 17 / 11 | 22 / 8 |
| Wallclock | 1920.7 s (32 min) | 1453.9 s |
| max_tokens cap | 8000 | 13000 |

**Read carefully — the raw pass@1 drop is from truncation, NOT model quality.** Per the repo's [`truncation_is_contamination`](https://github.com/canada-quant/dsv4-flash-nvfp4-fp8-mtp/blob/main/docs/findings/) note, raw pass@1 is contaminated when max_tokens differs between configs being compared. On 2× RTX PRO 6000 with `max_model_len=8192` (the memory-budget ceiling at TP=2 with MTP), we cap reasoning at ~8000 tokens vs the baseline's 13000 → 11 problems truncate mid-chain vs baseline's 8.

**MTP acceptance, the right model-quality metric, is identical (90.46% vs 90.61%).** Non-truncated rate is 17/19 = **89.5%** of completed-by-stop problems.

To match the baseline raw pass@1, deploy on 4× RTX PRO 6000 with TP=4 and `max_model_len=16384+` (the documented 2026-05-25 setup) — there's no model-side fix from a 2-GPU rig.

## AIME-2024 full 30 (c=1, thinking=high, **max_tokens=32000**, 2× RTX PRO 6000, MTP)

Re-run with `max_model_len=32768` + `max_tokens=32000` to remove truncation contamination. Uses `scripts/serve_rtx6000pro_tp2_32k.sh` (32K-context profile at seq=1).

| Metric | This run (TP=2, max=32K) | Baseline 2026-05-25 (TP=4, max=13K) |
|---|---|---|
| **Correct/30** | **24/30 (80.0%)** | 24/30 (80.0%) ✓ **exact match** |
| Errors | 0 | 0 |
| Finish: stop / length | 24 / 6 | 22 / 8 |
| **MTP acceptance** | **91.05%** | 90.61% (+0.44pt) ✓ |
| Wallclock | 2587.4 s (43 min) | 1453.9 s |
| max_tokens | 32000 | 13000 |

**The canonical, apples-apples number.** With sufficient reasoning budget, Card B on 2× RTX PRO 6000 + this Docker recipe **matches the documented baseline exactly** on AIME-2024 c=1 raw pass@1. 6 problems still hit `length` cap even at 32K — these are the genuinely-hard problems whose chains exceed 32K tokens (independent of the 2-GPU setup).

This invalidates the earlier 17/30 result (max_tokens=8000) as a quality reading — that was a context-budget artifact, not a regression. The 2× RTX PRO 6000 path is throughput-and-quality-equivalent to baseline for AIME-style reasoning when given enough max_tokens.

Wallclock is ~78% longer (43 min vs 24 min) because the 32K-context serve profile uses smaller `cudagraph_capture_sizes=[2]` only (vs the 8K profile's [1,2]) and the longer reasoning chains require more decode steps. Raw decode throughput at ~125 tok/s is similar.

## Thinking mode

✅ **Fully functional with the v3 image.** Earlier-session reports of "empty reasoning content" turned out to be a test-side field-name bug (we were reading `reasoning_content`, but vLLM 0.21+ renamed the field to `reasoning`). With the correct field name and `include_reasoning: true` in the request, all 3 effort levels return clean structured responses:

| Mode | reasoning chars | content chars | tokens | finish |
|---|---|---|---|---|
| `chat_template_kwargs.thinking=false` | 0 | 1840 | 815 | stop |
| `thinking=true, reasoning_effort=high` | 3776 | 359 | 1499 | stop |
| `thinking=true, reasoning_effort=max` | 6288 | 397 | 2824 | stop |

All three returned the correct final answer (`n=40` for the Euler prime-polynomial probe). `max` uses ~2× the reasoning tokens of `high`. No empty-body bug observed. The earlier-documented [vllm-project/vllm#34650](https://github.com/vllm-project/vllm/issues/34650) race may still exist as an edge case but is not the dominant behavior in our 2× RTX PRO 6000 + jasl@27fd665b deployment.

**API note**: client code expecting `reasoning_content` should switch to `reasoning`. Both `include_reasoning: true` in the request body and `--reasoning-parser deepseek_v4` on the server are required.
