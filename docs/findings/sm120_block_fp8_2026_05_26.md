# SM 12.0 block-FP8 dispatch + autotune investigation (2026-05-26)

> _Disclosure (per [`AGENTS.md`](https://github.com/vllm-project/vllm/blob/main/AGENTS.md#accountability)): drafted with AI assistance; all empirical claims below verified end-to-end on 4× RTX PRO 6000 Blackwell Server Edition (SM 12.0) against [`canada-quant/DeepSeek-V4-Flash-NVFP4-FP8-MTP`](https://huggingface.co/canada-quant/DeepSeek-V4-Flash-NVFP4-FP8-MTP). The human submitter (@pasta-paul / canada-quant) defends every claim._

> **Outcome:** Three patch sets together restore Card B (NVFP4-FP8-MTP) to coherent multi-step reasoning on RTX PRO 6000 Blackwell Server Edition. Two are our own filings ([`#43722`](https://github.com/vllm-project/vllm/pull/43722), [`#43723`](https://github.com/vllm-project/vllm/pull/43723)); one is an existing upstream PR ([`#41834`](https://github.com/vllm-project/vllm/pull/41834)). Two remaining caveats are upstream-blocked but documented.

## What we tested

- Hardware: 4× RTX PRO 6000 Blackwell Server Edition (SM 12.0, sm_120), PCIe interconnect
- Artifact: [`canada-quant/DeepSeek-V4-Flash-NVFP4-FP8-MTP`](https://huggingface.co/canada-quant/DeepSeek-V4-Flash-NVFP4-FP8-MTP) (freshly pulled from HF, not the now-deleted local snapshot)
- Base build: jasl/vllm@`a02a3778f` (ds4-sm120-preview-dev base) + the 3 still-open patches from [`#43564`](https://github.com/vllm-project/vllm/issues/43564)
- TP=4, `VLLM_TEST_FORCE_FP8_MARLIN=1`, MTP `num_speculative_tokens=1`

## What we observed before fixes

Engine crashes at `profile_run` with:
```
RuntimeError: b_scales dim 1 = 32 is not size_n = 1536
csrc/quantization/marlin/marlin.cu:701
```
Stack trace pins the layer to `compressor.fused_wqa_wkv` (DSv4 Q-A + KV-A projection, block-FP8 128×128 quantized).

## Root causes — three distinct bug classes

### 1. Marlin dispatch on block-FP8

`MarlinFP8ScaledMMLinearKernel.can_implement` returns True unconditionally. With `VLLM_TEST_FORCE_FP8_MARLIN=1` (required for NVFP4-MoE on SM 12.0), `is_supported` marks Marlin as supported on SM ≥ 89. Marlin is at priority 4 in `_POSSIBLE_FP8_BLOCK_KERNELS[CUDA]`, ahead of `TritonFp8BlockScaledMMKernel`. The dispatcher picks Marlin for block-FP8 layers. Marlin's per-channel scale layout cannot serve `weight_scale_inv` of shape `[N_blocks_n, N_blocks_k]` — the `marlin.cu:701` TORCH_CHECK fires.

**Fix:** [`#43722`](https://github.com/vllm-project/vllm/pull/43722) — refuse block-FP8 in `can_implement`. Dispatcher falls through to Triton.

### 2. `wo_a.weight_scale_inv` AttributeError on non-Marlin paths

Only `MarlinFP8.process_weights_after_loading` renames `weight_scale` → `weight_scale_inv`. Once `wo_a` routes to Triton (post-#43722), the direct read at `attention.py:330` AttributeErrors.

**Fix:** [`#43723`](https://github.com/vllm-project/vllm/pull/43723) — getattr fallback.

### 3. SM 12.0 has no working block-FP8 kernel except Triton, AND Triton's default autotune produces drifting outputs

After #43722 + #43723, dispatcher correctly selects Triton. But the dispatch list refusal pattern is:

```
FlashInferFp8DeepGEMMDynamicBlockScaledKernel: not available
DeepGemmFp8BlockScaledMMKernel: Only Hopper and Blackwell (SM 12.0 fails internal check)
CutlassFp8BlockScaledMMKernel: CUTLASS block-scaled FP8 not supported on SM12x
MarlinFP8ScaledMMLinearKernel: refused by #43722 (correct)
TritonFp8BlockScaledMMKernel: only viable kernel left
```

`TritonFp8BlockScaledMMKernel.is_supported` doesn't gate on compute capability beyond CUDA, but its inner kernel autotune at `vllm/model_executor/layers/quantization/utils/fp8_utils.py:718-806` is **looked up by device_name**. With no `NVIDIA_RTX_PRO_6000_Blackwell_Server_Edition` config present, default `BLOCK_SIZE_M=64, num_warps=4, num_stages=2` is used. That config is fundamentally mis-tuned for SM 12.0 — produces wrong outputs.

Symptom: `"What is 2+2?"` → `" 4*4=16, so 4*4=16, so 4*4=16, so..."` (degenerate token loop from token 1).

**Fix:** [`#41834`](https://github.com/vllm-project/vllm/pull/41834) — jasl's open PR ships 6 linear-side + 10 MoE-side tuned configs for RTX PRO 6000 Server Edition. With them:

- `"What is 2+2?"` → `" 4\n\nWhat is your name? Answer: John Smith\nWhat is your quest?..."` (correct + coherent)
- `"Compute 17 * 23"` → `"17*23 = 17*20 + 17*3 = 340 + 51 = 391"` (correct, with proper chain of thought)
- MTP acceptance: 13.65% (pre-tuned) → 34.66% (post-tuned)

## Remaining caveats

### A. `--reasoning-parser deepseek_v4` × MTP × thinking-mode race ([`#34650`](https://github.com/vllm-project/vllm/issues/34650))

`chat_template_kwargs={"thinking":true,"reasoning_effort":"high"}` requests complete with `completion_tokens > 200` but both `reasoning_content==''` and `content==''`. The model generates output; the structured-output `should_advance` race at `vllm/v1/structured_output/__init__.py:324-328` misses the `</think>` boundary, and the entire response gets dropped.

Repro confirmed and posted to #34650 at https://github.com/vllm-project/vllm/issues/34650#issuecomment-4550197205.

**Workaround until upstream fixes:** drop `--reasoning-parser deepseek_v4` or use `thinking=false`. With thinking off, single-question math (`"Find f(5)"`) produces correct chain of thought.

### B. Autotune coverage gap in #41834

The 6 linear-side configs cover N ∈ {1536, 2048, 4096, 8192, 16384} × K ∈ {1024, 4096}. DSv4-Flash uses additional shapes (e.g. `wo_a` N=1024/K=512) that fall back to default autotune. Symptom: coherence holds for ~100-150 generated tokens, then mild drift starts. Filed as a coverage-ask on PR #41834.

```bash
    --tensor-parallel-size 4 \
    --speculative-config '{"method":"mtp","num_speculative_tokens":1}' \
    --kv-cache-dtype fp8 --max-model-len 32768 \
    --trust-remote-code

curl http://localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"DSV4-NVFP4-FP8-MTP","messages":[{"role":"user","content":"Compute 17 * 23. Show your work."}],"max_tokens":400,"temperature":0}'
# Expect: "17*23 = 17*20 + 17*3 = 340 + 51 = 391" (correct chain of thought)
```
